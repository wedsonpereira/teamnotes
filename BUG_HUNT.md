# TeamNotes Bug Hunt

Mission: find, document, prioritize, and fix bugs feature by feature.

## Status Legend

- `Open`: confirmed issue, not fixed yet
- `Fixed`: patched in the current worktree
- `Watching`: possible risk, needs reproduction or more evidence
- `No issue found`: inspected and no actionable bug found

## Phase 1: Room Entry

### Feature 1: Create Room

Expected behavior:

- User opens Create Room from the landing page.
- User enters username, email, optional Room ID, password, and confirm password.
- Form validation matches backend validation.
- API creates or reuses the user safely.
- API creates a room, admin membership, and room session.
- Success screen shows Room ID and Room Key.
- Enter Room stores a valid client session and navigates to the room.

Test checklist:

- [x] Create with autogenerated Room ID: audited UI/API path.
- [x] Create with custom valid Room ID: audited UI/API path.
- [x] Reject invalid custom Room ID: API validates `^[A-Za-z0-9_-]{4,32}$`.
- [x] Reject duplicate custom Room ID: API returns 409 on `roomCode` conflict.
- [x] Reject invalid email: API validates email format.
- [x] Reject password shorter than backend minimum: UI and API both require 8 characters.
- [x] Reject mismatched password confirmation: UI blocks submit.
- [x] Enter created room from success screen: session now uses API-returned username.
- [x] Confirm room owner appears as username, not first/last name: API returns username and empty lastName.

Findings:

| ID | Status | Severity | Finding | Notes |
| --- | --- | --- | --- | --- |
| CR-001 | Fixed | Medium | Existing-email create could mix identities. | API did not overwrite existing users, but room name used the newly typed username while the returned user kept the existing username. Room name and create response now use the effective stored username. |
| CR-002 | Fixed | Medium | Success-screen Enter Room used local username instead of API username. | Client session now prefers `success.username`, preventing local/session display drift. |
| CR-003 | Fixed | Low | Username had no create-form length guard. | UI and API now cap usernames at 48 characters and trim whitespace. |
| CR-004 | No issue found | Low | Password validation mismatch. | Current UI and API both use 8-72 characters. |

### Feature 2: Join Room / Already-Member Access

Expected behavior:

- Email is the identity key for room membership.
- If the email already belongs to an approved member in that room, the user enters immediately after providing the correct Room ID and Room Key.
- If the email belongs to a pending member, the user stays pending unless an admin invite exists.
- If the email belongs to a rejected member, the user is blocked unless an admin invite exists.
- If the email is not in the room member list, a new join request is created for admin approval.
- If an admin removed the member from the room, that email is no longer treated as approved.

Test checklist:

- [x] Existing approved email joins without admin approval.
- [x] Existing pending email remains pending.
- [x] Existing rejected email remains blocked.
- [x] New email creates a pending join request.
- [x] Removed member email creates a fresh pending request instead of auto-entering.
- [x] Client session uses server-returned username after auto-entry.

Findings:

| ID | Status | Severity | Finding | Notes |
| --- | --- | --- | --- | --- |
| JR-001 | Fixed | High | Join flow checked membership after form-driven user mutation. | Membership is now checked first by room plus email, so an already-approved email enters immediately and keeps the stored room identity. |
| JR-002 | Fixed | Medium | Approved join response could be stored with the typed username on the client. | Landing page now prefers `data.username` when entering after join. |

### Feature 3: Admin Join Request Notification

Expected behavior:

- When a new user requests to join a room, connected admins receive a realtime alert.
- The admin sees a visible in-app notification while working in the editor.
- A short sound plays with the notification when the browser allows audio.
- The notification lets the admin jump to the pending request list.
- Member list still refreshes in realtime after the request is created.

Test checklist:

- [x] New pending join request emits a dedicated realtime notification event.
- [x] Admin room clients listen for `join-request`.
- [x] Admin sees a toast with the requester name.
- [x] Admin hears a short sound cue when browser audio is available.
- [x] Toast Review action opens the members sidebar.
- [x] Existing members/pending/rejected repeat attempts do not create duplicate request sounds.

Findings:

| ID | Status | Severity | Finding | Notes |
| --- | --- | --- | --- | --- |
| JN-001 | Fixed | Medium | Admins only got silent member refreshes for new join requests. | Join API now emits `join-request` when a new pending membership is created. |
| JN-002 | Fixed | Medium | No visible or audible admin-side request alert. | Room page now shows an admin toast, plays a short sound, and opens the sidebar from Review. |

### Feature 4: View-Only Access Enforcement

Expected behavior:

- Newly joined or invited non-admin members start as view-only.
- View-only members cannot save document changes.
- View-only members cannot create, rename, reorder, or delete pages.
- View-only members cannot bypass restrictions through realtime Yjs socket updates.
- Admins can later change a member from View to Edit.

Test checklist:

- [x] Socket room access defaults missing/legacy member access to `VIEW`.
- [x] Socket Yjs update gate only permits admins or explicit `EDIT` members.
- [x] Page mutation APIs require explicit `EDIT` access.
- [x] Save API requires explicit `EDIT` access.
- [x] Admin member access select defaults missing access to View.
- [x] Admin approval forces pending members to View before they can enter.

Findings:

| ID | Status | Severity | Finding | Notes |
| --- | --- | --- | --- | --- |
| VA-001 | Fixed | High | Realtime and page/save permission fallbacks failed open to edit. | Socket, page, and save access checks now require explicit `EDIT`; missing access falls back to `VIEW`. |
| VA-002 | Fixed | Medium | Approving an older pending request could preserve stored Edit access. | Admin approval now sets approved members to `VIEW`; admins can switch them to `EDIT` afterward. |

### Feature 5: Room Settings Refresh

Expected behavior:

- Admin can toggle member-list visibility.
- Non-admin clients refresh their sidebar when room settings change.
- Settings propagation does not depend on the admin tab's socket emit succeeding.

Test checklist:

- [x] Settings API verifies admin session before updating.
- [x] Settings API emits `settings-refresh` after a successful update.
- [x] Sidebar listens for `settings-refresh` and reloads room/member data.

Findings:

| ID | Status | Severity | Finding | Notes |
| --- | --- | --- | --- | --- |
| RS-001 | Fixed | Medium | Settings refresh depended on a follow-up client socket emit. | Settings API now emits `settings-refresh` directly after saving, so other clients update even if the admin socket is unavailable. |
